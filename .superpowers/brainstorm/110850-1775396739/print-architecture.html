Print Architecture — 3 approaches

All three fix the print bugs and add the digital docket modal. They differ only in how the admin/central printer terminal is identified.

A

Role-based Recommended

Waiter (org:member) → Queue job in Appwrite → Admin (org:admin) PrintBridge processes it → USB Thermal Printer

How it works: PrintBridge reads the Clerk role on startup. If org:admin → process jobs. If org:member → queue only, never process.

✓ Zero setup — roles already exist in Clerk
✓ Any admin tab = automatic printer terminal
⚠ Admin must keep a browser tab open with POS
B

Terminal Registration

Any waiter → Queue job with targetTerminalId → Registered terminal (any account) processes it → USB Printer

Admin goes to POS Settings → "Register this device as printer terminal". Saves a UUID to Clerk org publicMetadata. PrintBridge matches its own UUID against the org setting.

✓ Explicit — any device can be the printer
✓ Survives account changes (role-independent)
⚠ Requires one-time setup UI in POS settings
✗ Slightly more complex — extra Clerk metadata read on job creation
C

Always Queue, Admin Processes All

Any waiter → Queue job (no targeting) → First PrintBridge with USB config processes it → USB Printer

Simplest. All devices queue jobs via Appwrite. Only the admin device has a USB printer configured in localStorage. It naturally becomes the sole processor — no role check needed.

✓ Fewest moving parts
✓ No Clerk metadata reads
⚠ Race condition if two devices accidentally have a printer configured
✗ No explicit guarantee of which terminal prints
